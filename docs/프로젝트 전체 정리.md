# Adaptive Kiosk - 프로젝트 전체 정리

이 문서는 현재 백엔드 코드의 모든 내용을 정리한 것으로, 이 문서만으로 프로젝트 전체 구조와 각 파일의 역할, 앞으로 채워야 할 부분을 파악할 수 있다.

---

## 1. 프로젝트 개요

디지털 취약계층(고령층 등)이 키오스크를 더 쉽게 사용할 수 있도록 돕는 지능형 음료 주문 키오스크 시스템의 **FastAPI 백엔드**.

### 핵심 기능 6가지

| # | 기능 | 담당 모듈 | 현재 상태 |
|---|------|-----------|-----------|
| 1 | 비전 기반 UI 자동 전환 (연령 추정 → 간편모드) | vision_service, vision endpoint | 구조 완성, 비전 모듈 연동 필요 |
| 2 | 음성 주문 (TTS/STT + AI 대화) | voice_service, voice endpoint | 구조 완성, STT/TTS 프런트 연동 필요 |
| 3 | 음료 추천 (선호 계열 + CF 발견형) | recommendation_service, recommendation endpoint | 구조 완성, CF 모델 교체 필요 |
| 4 | 주문 관리 | order endpoint, crud | 완성 |
| 5 | 로그 기반 정량 평가 | analytics_service, analytics endpoint | 완성 (Basic Auth 보호) |
| 6 | 멀티 키오스크 관리 | kiosk endpoint, X-API-Key 인증 | 완성 |

## 2. 기술 스택

| 분류 | 기술 | 비고 |
|------|------|------|
| Framework | FastAPI (전체 async) | lifespan으로 DB 초기화 |
| ORM | SQLAlchemy 2.0 async | aiomysql 드라이버 |
| DB | MySQL | 테이블 자동 생성 (create_all) |
| 인증 | HTTP Basic Auth (Swagger/관리자) + API Key (키오스크) | JWT 미사용 |
| AI | Google Gemini Flash Lite (google-genai) | 음성 주문 시나리오 처리 |
| 패키지 관리 | uv (pyproject.toml) | |

## 4. core/ - 핵심 설정

### core/config.py
- `pydantic-settings`의 `BaseSettings`로 `.env` 자동 로드
- 설정값: `KIOSK_USERNAME`, `KIOSK_PASSWORD`, `DATABASE_CONN`, `GENAI_API_KEY`
- `settings = Settings()` 인스턴스를 export하여 전체 프로젝트에서 사용

### core/database.py
- `create_async_engine` + `AsyncAdaptedQueuePool` 사용 (pool_size=32, max_overflow=20)
- `_engine`, `_async_session_factory`를 모듈 내부 변수로 관리, `get_engine()`, `get_session_factory()` 함수로 접근
- `initialize_connection_pool()`: 서버 시작 시 1회 호출, `SELECT 1`로 연결 테스트
- `get_db()`: FastAPI Depends용 async 세션 제너레이터
- `DatabaseConnectionError`: DB 관련 커스텀 예외

### core/security.py
- **HTTP Basic Auth**: `http_basic`, `verify_credentials`
- Swagger docs 보호 + 관리자 전용 엔드포인트(analytics, kiosk register/list) 보호용
- `.env`의 `KIOSK_USERNAME` / `KIOSK_PASSWORD`로 인증

## 5. 인증 구조

| 대상 | 인증 방식 | 설명 |
|------|-----------|------|
| Swagger 문서 (`/docs`, `/redoc`) | HTTP Basic Auth | KIOSK_USERNAME/PASSWORD. main.py 미들웨어에서 처리 |

| 공개 API (health, kiosk verify, vision, voice 등) | 인증 없음 | |

## 7. services/ - 비즈니스 로직

### vision_service.py - 간편모드 판단
- `should_use_simple_mode(vision_data)`: 연령대가 60대/70대이상이고 confidence >= 0.7이면 True
- **향후**: 임계값 조정, 혼란 감지 결과 연동

### recommendation_service.py - 추천 생성
- `get_recommendations(db, preferred_category, gender, age_group)` → `{ your_picks, discovery }`
- **your_picks**: 선호 카테고리 내 메뉴 목록 (DB 조회)
- **discovery**: 선호 카테고리가 아닌 메뉴 중 최대 5개 (현재 단순 필터링)
- **향후**: User-Based CF 모델로 교체

### analytics_service.py - 통계 집계
- `get_session_analytics(db)` → 총 세션 수, 간편모드 전환율, 도움 트리거 수
- `get_recommendation_analytics(db)` → 추천 노출 수, 클릭률, 주문 전환율
- `get_order_analytics(db)` → 총 주문 수, 평균 주문 시간, 추천 경유 비율
- **향후**: 키오스크별/기간별 필터링

### voice_service.py - 음성 주문 처리
- **2단계 처리**:
  1. `match_pattern(stt_text)`: 정규식 패턴 매칭 → 6가지 intent
  2. `ask_gemini(stt_text, history)`: 패턴 매칭 실패 시 Gemini Flash Lite 호출
- `process_voice_input(stt_text, history)` → `{ intent, matched_by, response_text, requires_followup }`
- **패턴 매칭 intent**: order_confirm, order_decline, help, cancel, menu_mention, confirm
- **향후**: 메뉴 이름 DB 연동, 수량/옵션 파싱, matched_menu_ids 추가

## 9. 데이터 흐름

### 키오스크 등록 및 인증

```
관리자        → POST /kiosks/register (Basic Auth) → API 키 발급
키오스크 시작 → POST /kiosks/verify                → 기기 확인 + 정보 반환
이후 세션    → POST /sessions (X-API-Key)          → kiosk_id 자동 기록
```

### 일반 주문 흐름

```
사용자 접근 → POST /sessions (X-API-Key) → 세션 생성
                              ↓
비전 분석   → POST /vision/{session_id}  → 간편모드 판단
                              ↓
                    ┌── 일반 모드 → 터치 주문
                    └── 간편 모드 → 음성 주문 or 간편 터치 주문
                              ↓
추천 요청   → POST /recommendations/request → your_picks + discovery
                              ↓
주문 완료   → POST /orders → 주문 저장
                              ↓
세션 종료   → POST /sessions/{id}/end
```

### 음성 주문 흐름

```
간편모드 진입 → POST /voice/{id}/start     TTS: "주문하시겠습니까?"
      ↓
사용자 음성  → POST /voice/process         STT 텍스트 수신
      ↓
            ┌── 패턴 매칭 성공 → 즉시 응답 (matched_by: "pattern")
            └── 실패 → Gemini Flash Lite (matched_by: "gemini")
      ↓
TTS 응답    → requires_followup=true면 반복, false면 종료
```

## 13. 향후 구현/수정이 필요한 부분

### 비전 모듈 연동
- 현재: POST /vision/{session_id}로 결과를 받는 구조만 존재
- **필요**: 비전 모듈(OpenCV + YOLOv8 + DeepFace)을 서버에서 실행, 프런트에서 프레임 전송
- **파일 위치**: `services/vision_service.py` + 새 endpoint 추가

### 추천 시스템 고도화
- 현재: 카테고리별 단순 필터링
- **필요**: User-Based CF 모델 (KNN 기반), 트렌드/시간대 보정
- **파일 위치**: `services/recommendation_service.py`의 `get_recommendations()` 교체

### 음성→주문 연결
- 현재: voice 응답에 menu_id가 없어서 음성 주문 → 실제 주문 생성 연결이 안 됨
- **필요**: voice 응답에 matched_menu_ids 추가, Gemini 프롬프트에 메뉴 목록 주입
- **파일 위치**: `schemas/voice.py`, `services/voice_service.py`

### 세션 abandoned 처리
- 현재: 세션 종료 API가 호출되지 않으면 영원히 active
- **필요**: 프런트 무활동 타임아웃 + 백엔드 주기적 정리
- **파일 위치**: 프런트엔드 + `main.py` lifespan에 백그라운드 태스크

### analytics 필터링
- 현재: 전체 통계만 반환
- **필요**: kiosk_id별, 기간별 필터링 파라미터
- **파일 위치**: `services/analytics_service.py`, `api/v1/endpoints/analytics.py`

### 메뉴 관리 API
- 현재: 조회만 가능 (GET)
- **필요**: 메뉴/카테고리 등록/수정/삭제 (관리자용, Basic Auth)
- **파일 위치**: `crud/menu.py`, `api/v1/endpoints/menu.py`

### 메뉴 시드 데이터
- 현재: DB에 메뉴가 없으면 추천/주문 테스트 불가
- **필요**: 초기 카테고리/메뉴 데이터를 넣는 스크립트
- **파일 위치**: `scripts/seed_menu.py` (새 파일)

## 14. 계층 구조 요약

```
main.py (앱 진입)
  ↓
api/v1/endpoints/ (HTTP 요청 수신)
  ↓
services/ (판단 로직: 간편모드, 추천, 음성 AI, 통계)
  ↓
crud/ (순수 DB 읽기/쓰기)
  ↓
models/ (테이블 구조)

schemas/ (입출력 데이터 검증, 모든 계층에서 사용)
core/ (설정, DB 연결, 보안 - 모든 계층의 기반)
```

새 기능을 추가할 때는: **model → schema → crud → service (필요 시) → endpoint → router 등록** 순으로 작성.
