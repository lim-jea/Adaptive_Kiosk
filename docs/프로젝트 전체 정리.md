# Adaptive Kiosk - 프로젝트 전체 정리

이 문서는 현재 백엔드 코드의 모든 내용을 정리한 것이다.

---

## 1. 프로젝트 개요

디지털 취약계층(고령층 등)이 키오스크를 더 쉽게 사용할 수 있도록 돕는 지능형 음료 주문 키오스크 시스템의 **FastAPI 백엔드**.

### 개발 단계

```
1단계 (현재): 기본 키오스크 기능
              메뉴 조회 → 옵션 확인 → 주문 확정

2단계: 지능형 기능
       비전 UI 전환 / 추천 / 음성 주문

3단계: 고도화
       CF 추천 모델 / 세션 로그 분석 / 관리자 대시보드
```

### 핵심 기능

| # | 기능 | 상태 |
|---|------|------|
| 1 | 기본 주문 (메뉴 조회 → 옵션 → 주문) | **1단계 완성** |
| 2 | 멀티 키오스크 관리 (X-API-Key) | **완성** |
| 3 | 비전 기반 UI 전환 | 2단계 (코드 있음, router 비활성) |
| 4 | 음료 추천 (플로팅 패널) | 2단계 (코드 있음, router 비활성) |
| 5 | 음성 주문 (패턴 매칭 + Gemini) | 2단계 (코드 있음, router 비활성) |
| 6 | 로그 기반 정량 평가 | **완성** (관리자 Basic Auth) |

| 분류 | 기술 |
|------|------|

| ORM | SQLAlchemy 2.0 async (aiomysql) |

## 6. 기본 키오스크 API (1단계 — 현재 활성)

### 모든 Request는 Pydantic 스키마로 통제됨

### Kiosk
| 메서드 | 경로 | Request | 인증 | 설명 |
|--------|------|---------|------|------|
| POST | /kiosks/register | KioskRegisterRequest | Basic Auth | 키오스크 등록 |
| POST | /kiosks/verify | KioskVerifyRequest | 없음 | 기기 확인 |
| GET | /kiosks | - | Basic Auth | 키오스크 목록 |

### Session
| 메서드 | 경로 | Request | 인증 | 설명 |
|--------|------|---------|------|------|
| POST | /sessions | - | X-API-Key | 세션 시작 → session_uuid 반환 |
| GET | /sessions/{session_uuid} | SessionGetRequest | 없음 | 세션 조회 |
| POST | /sessions/{session_uuid}/end | SessionEndRequest | 없음 | 세션 종료 |

### Menu
| 메서드 | 경로 | Request | 인증 | 설명 |
|--------|------|---------|------|------|
| GET | /menus | MenuListRequest | 없음 | 메뉴 목록 (?category_id=) |

## 7. Request 스키마 상세

### KioskRegisterRequest
- `name`: str (1~100자)
- `location`: Optional[str] (최대 200자)

### KioskVerifyRequest
- `api_key`: str (정확히 64자)

### SessionEndRequest
- `reason`: Literal["completed", "timeout", "cancelled"]

### MenuListRequest
- `category_id`: Optional[int] (양수)

### MenuDetailRequest
- `menu_id`: int (양수)

### OrderCreateRequest
- `session_uuid`: str (정확히 32자)
- `items`: List[OrderItemRequest] (최소 1개)
- `used_recommendation`: bool

### OrderItemRequest
- `menu_id`: int (양수)
- `quantity`: int (1~99)
- `unit_price`: int (0 이상, 서버에서 재검증)
- `from_recommendation`: bool
- `selected_options`: List[SelectedOptionRequest]

### SelectedOptionRequest
- `option_item_id`: int (양수)

## 8. 주문 흐름 + 가격 검증

```
프런트:
  unit_price = menu.price + Σ(선택 option_item.extra_price)
  total      = Σ(unit_price × quantity)
  → 장바구니 화면에 실시간 표시

서버 (POST /orders):
  1. session_uuid → 내부 session_id 변환
  2. 각 아이템의 menu_id 존재/판매중 검증
  3. 각 option_item_id 존재 검증
  4. unit_price 서버 사이드 독립 재계산
  5. total_price = Σ(서버 unit_price × quantity)
  6. 옵션 스냅샷 저장 (이름, 추가금)
  7. 응답에 menu_name, option_name 포함
```

### 옵션 그룹 (6개)
- 사이즈: Tall(+0) / Grande(+500) / Venti(+1000) — 필수
- 온도: HOT / ICE — 필수 (temp=both인 메뉴에)
- 샷 추가: +500 — 선택 (최대 3)
- 시럽: 바닐라/헤이즐넛/카라멜 (+300) — 선택 (최대 2)
- 휘핑크림: +500 — 선택
- 당도: 기본/덜 달게/더 달게 — 필수 (달콤한 음료에)

## 12. 향후 작업

### 2단계 — 지능형 기능
- 비전 모듈 서버 연동 (POST /vision/analyze)
- 추천 트리거 3단계 (default → tab_dwell → menu_select)
- 음성→메뉴 파싱 (matched_menu_ids)
- 세션 abandoned 자동 처리
- session_events 테이블 (UX 타임라인 로그)

### 3단계 — 고도화
- CF 추천 모델 (KNN + OpenSurvey 데이터)
- analytics 기간/키오스크별 필터
- 관리자 메뉴/옵션 CRUD API
- 관리자 JWT 인증 (admins 테이블)

## 13. 계층 구조

```
main.py (앱 진입)
  ↓
api/v1/endpoints/ (Request 스키마로 입력 통제)
  ↓
services/ (비즈니스 로직)
  ↓
crud/ (DB 접근)
  ↓
models/ (테이블 구조)

schemas/ (Request/Response 스키마 — 모든 계층에서 사용)
core/ (설정, DB, 보안)
scripts/ (시드 데이터)
```

새 기능 추가 순서: **model → schema → crud → service → endpoint → router 등록**
